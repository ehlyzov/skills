# Универсальный чек-лист критика

Этот файл используется всеми тремя критиками (scenario-critic, plan-critic, hardening-auditor) и независимым верификатором как общий справочник того, что нужно проверять. Каждый агент-критик читает свою специализированную часть.

## Уровни проверки

### Уровень 1 — Структурная корректность

- Все обязательные секции на месте (для сценария — 9 разделов; для T-задачи — 9 полей; для overview — 10 секций).
- Подсекции User flow присутствуют все три (Основной / Альтернативные / Ошибочные).
- Чекбоксы Status имеют правильный формат `- **Status:** - [ ]` или `- **Status:** - [x]`.
- Markdown ссылки валидны (нет битых якорей).

### Уровень 2 — Связность

- Cross-references двусторонни. Если сценарий A упоминает B в шапке, то B упоминает A.
- Индекс в overview содержит все файлы сценариев.
- Каждая T-задача `Depends on:` указывает на существующие задачи.
- Каждая H-задача `Depends on: TXX` — на реальный T в implementation plan.

### Уровень 3 — Непротиворечивость

- Числа в overview Per-scenario метриках совпадают с числами в самих сценариях.
- Цели из overview покрыты в acceptance/NFR ровно одного сценария с тем же числовым ориентиром.
- Persona-имена единообразны во всём корпусе (case-sensitive).
- Один и тот же FR/NFR не описан в двух сценариях с разными формулировками или разными числами.
- Acceptance criterion проверяется хотя бы одним пунктом Test plan.

### Уровень 4 — Замкнутость UX

- Каждый user journey имеет начало и конец.
- Раздел «Дополнительные сценарии и связь с продуктом» (после Phase 2) явно описывает откуда пользователь пришёл и куда уходит.
- Нет «висящих» сценариев, ссылающихся на несуществующее.
- Если scenario X говорит «cleanup делается в [Y]», то Y действительно описывает cleanup.

### Уровень 5 — Реалистичность

- Числовые ориентиры (latency, time-to-X, percentage) не фантастика для текущего стека.
- Inline-код в Steps T-задачи синтаксически корректен.
- Verify-команды используют реальные модули и test runners.
- Файлы и line-anchors, упомянутые в Read first / Modify, существуют (если код есть).

### Уровень 6 — Безопасность контракта

- Stable API endpoints (`/api/debug/*`, `/api/agent/*`, и т.п.) — additive only в minor.
- Никакая T-задача не правит acceptance criteria сценариев.
- Никакая H-задача не вводит новой функциональности (только инвариант / тест / политика).
- mutating endpoints явно различены от read-only — где это требуется по сценарию.

### Уровень 7 — Человеческие решения и стиль

- Нетривиальные продуктовые решения отражены в `docs/product/decisions.md`.
- Планы не опираются на решения со статусом `proposed`.
- Агентская рекомендация не записана как человеческое решение без approval source.
- Продуктовый текст написан цельно по-русски; английский остался только для кода, команд, путей, API и собственных названий.
- Stakeholder-facing PDF не содержит T/H-задач и worker-инструкций без явного запроса.

## Формат отчёта критика

Каждый критик возвращает:

```markdown
### Вердикт
**Одно предложение:** «Всё хорошо» или «Нужны уточнения».

### Если «всё хорошо»
- 3-5 сильных сторон.
- 1-2 микро-замечания (косметика, не блокеры).
- Закрой отчёт.

### Если «нужны уточнения»
По каждому замечанию:
- file_path:line (или раздел)
- Что не так (одна фраза)
- Минимальная правка (одна строка)

Сортируй critical → major → minor.

### Открытые вопросы
0-3 если есть реальные смысловые развилки.
```

**≤ 1000 слов на отчёт.** Конкретность обязательна (file:line).

## Цикл итерации

1. Применить все critical и major правки автоматически (где не нужны решения автора).
2. Открытые вопросы — задать пользователю через AskUserQuestion (max 4 вопроса в одном вызове).
3. Применить ответы, прогнать критика снова.
4. Если 3 раунда подряд критик возвращает «всё хорошо» с разными микро-замечаниями — считать фазу закрытой; накопленные микро-замечания применить как косметику без нового раунда.
5. Если критик 5 раундов подряд возвращает new critical findings — звать пользователя на консультацию (вероятно, есть структурная проблема).
