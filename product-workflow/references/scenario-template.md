# Шаблон одного пользовательского сценария

Каждый файл `docs/product/scenarios/NN-<slug>.md` пишется по этому шаблону. **Все 9 разделов обязательны**, ни один не сокращается. У User flow строго три подсекции (Основной / Альтернативные / Ошибочные). Test plan — четыре уровня (Unit / Integration / E2E / Migration / rollback) с пометкой N/A где неприменимо.

```markdown
# NN — <Название сценария>

- **Persona:** <одна-три персоны, lowercase if compound — например «dev-on-call», совпадает с overview персонами>.
- **Статус:** current | growth.
- **UI:** <ссылки markdown на UI-страницы, относительные пути, например `[/runtime](../../../launchpad-ui/src/routes/RuntimeDashboardPage.tsx)`>.
- **Backend:** <ссылки на endpoints / сервисы, с line-anchor где осмысленно>.
- **Связанные сценарии:** <ссылки на 2-4 соседних сценария в формате `[NN <Название>](NN-slug.md)`, симметрично — если X ссылается на Y, в Y должна быть ссылка на X>.

## Проблема

5-10 строк или маркеров: какая боль у пользователя/бизнеса. Конкретно, с примерами. Если есть гэп-fit между текущей реальностью и желаемой — обозначить.

## Цель

1-3 предложения: какой результат хотим получить. Измеримые ориентиры (время, %, штуки). Если сценарий growth — отметить «целевой результат после реализации MVP».

## Не входит в сценарий

3-6 пунктов: что **не делаем** в этом сценарии. Чтобы scope был чётким и smooth-транзишен в соседние сценарии.

## Пользовательский сценарий

### Основной сценарий

Пронумерованный список шагов (5-12 шагов). Конкретно: какие UI-кнопки, какие API-вызовы, какие переходы между экранами/сервисами. Ссылки на файлы кода — допустимы и приветствуются.

### Альтернативные сценарии

Маркированный список A1, A2, ... — реалистичные альтернативы основному пути (через CLI, при пустом профиле, batch-режим, etc.).

### Ошибочные сценарии

Маркированный список E1, E2, ... — что пойдёт не так и как UI/backend это обработает. Не «теоретические» — реальные ошибки.

## Функциональные требования

FR1, FR2, ... — что система **должна** делать. Императивно. Каждый FR — одно конкретное поведение. Должно соответствовать User flow и быть проверяемым тестом.

## Нефункциональные требования

NFR1, NFR2, ... — Latency, consistency, permissions, audit, observability, scalability. С числами где применимо. Если сценарий привязан к Goal главного документа — пометить «(GX)» рядом с relevant NFR.

## Архитектурные ограничения

AC1, AC2, ... — какие слои можно/нельзя менять, какие API нельзя ломать, какие зависимости запрещены. Это инварианты кода, не процессы.

## Критерии приёмки

AC-1, AC-2, ... — проверяемые критерии готовности. Каждый — измерим через тест из Test plan ниже. Никаких «работает хорошо» / «удобно» — только конкретные числа и явные shape-проверки.

## План проверки

- **Unit:**
  - <тесты на функции/классы>
- **Integration:**
  - <тесты на API endpoints, запись в БД, etc.>
- **E2E:**
  - <тесты с реальным стендом>
- **Migration / rollback:**
  - <если применимо; иначе явная пометка `N/A — функциональность не меняет схему`>
```

## Опциональный раздел для growth-сценариев

Для сценариев со `Статус: growth` добавить **в самом конце** секцию gap-анализа:

```markdown
## Дополнительно: gap analysis vs текущее состояние

| Что есть сегодня | Что нужно для <фичи> | Объём работы |
| --- | --- | --- |
| ... | ... | low/medium/high |
```

## Правила нумерации

- Файлы — `01-<slug>.md` ... `NN-<slug>.md`. Slug — kebab-case английский.
- Внутри одного файла FR/NFR/AC/E нумеруются с 1. Между файлами — независимо.
- Goals главного документа — `G1..GN` (общие), цитируются в сценариях как `(GX)`.

## Что критик проверяет в этом шаблоне

См. [critic-checklist.md](critic-checklist.md). Главное:

1. Все 9 разделов присутствуют (`grep -c '^## ' file.md` ≥ 9). Предпочтительные заголовки — русские; `verify_artifacts.py` временно принимает старые английские заголовки для обратной совместимости.
2. Все три подсекции User flow присутствуют (`grep '^### Основной\|^### Альтернативные\|^### Ошибочные'`).
3. Persona в шапке = persona в overview index.
4. Cross-refs в шапке двусторонни с другими сценариями.
5. Каждый AC покрыт хотя бы одним пунктом Test plan.
6. Числа в AC и NFR не противоречат overview Goals.
7. Все file:line references указывают на существующие файлы (если в репозитории есть код).
