# Шаблон stakeholder-facing PDF

PDF — продуктовый документ для стейкхолдеров. Он объясняет задачу, выбранное решение и ожидаемый результат. Он не является планом работ для worker-а.

## Включать по умолчанию

1. **Продуктовая задача** — какая проблема решается и для кого.
2. **Контекст и ограничения** — факты из discovery, текущая реальность, важные ограничения.
3. **Выбранное решение** — что именно делаем.
4. **Почему выбран этот вариант** — trade-offs из `docs/product/decisions.md`, включая отвергнутые альтернативы.
5. **Сценарии пользователя** — current/growth сценарии, но без command-level задач.
6. **Метрики успеха** — цели и измеримые ориентиры.
7. **Риски и ограничения** — что сознательно не входит в scope.
8. **Итоговая независимая проверка** — краткий verdict из `docs/product/validation/verdict.md`.

## Не включать по умолчанию

- `docs/plans/*implementation-plan*.md`;
- `docs/plans/*hardening-plan*.md`;
- задачи `TXX` / `HXX`;
- worker-инструкции, команды запуска задач, commit conventions;
- длинные технические чек-листы, если они не объясняют продуктовый выбор.

Если пользователь просит внутренний инженерный PDF, можно собрать с `INCLUDE_ENGINEERING_PLANS=1`, но в ответе явно сказать, что это уже не stakeholder-facing документ.

## Стиль

- Язык: русский.
- Английский оставлять только для кода, API, команд, путей, названий библиотек и собственных имён.
- Заголовки продуктовых разделов писать по-русски.
- Не использовать смесь вроде «Phase», «Goal», «Non-goal», «Product map» в PDF, если это не часть исходного API или названия.
- Каждая альтернатива должна быть объяснена спокойно: почему не выбрана, какой риск или стоимость у неё выше.
